Offline voucher protocol v1

A system direction for replacing offline signed transaction handoff with escrow-backed voucher claims that can be checked locally and settled later.

Back to Research Related Product Note
Goal

Bounded offline trust instead of pretending offline finality exists.

The main idea is to stop transporting fully signed blockchain transactions offline and instead move toward escrow-backed voucher claims. That changes the trust model. The receiver is no longer trusting a transport payload alone. The receiver is checking a claim against a committed escrow state.

This does not try to guarantee instant offline finality. It aims for safer deferred settlement with bounded loss and clearer verification rules.

Core model

An escrow commits to a limited voucher set.

In the proposed model, an escrow locks funds on-chain and commits to a bounded set of vouchers. Each voucher carries the amount, expiry window, nonce, and optional receiver binding. The receiver verifies the voucher locally using proof data tied back to the escrow commitment.

  • Sender signature must be valid.
  • Voucher inclusion proof must match the committed root.
  • Escrow proof must match finalized state.
  • Expiry and duplicate protections must still pass locally.
System implications

The backend helps, but it is not the trust anchor.

The worker can index escrow commitments, serve proof bundles, relay messages, and deduplicate claims, but it is not the party that makes a bad voucher trustworthy. The security comes from the proofs, signature validity, expiry windows, and deterministic claim handling.

That separation keeps the offline payment model aligned with BitSend’s local-first direction instead of moving the trust burden entirely into a service layer.
